# 実装方針書

この文書は、本リポジトリにおける今後の設計、実装、資料整理の既定方針である。
ユーザーから明示的な上書き指示がない限り、以後の判断はこの文書を基準に行う。
方向性を変える場合は、コードだけを先に変えるのではなく、この文書も更新して整合を取る。

この文書でいう `surrogate` とは、厳密な物理式や詳細モデルをそのまま解く代わりに、
似た挙動を扱いやすく再現するために用いる近似モデル、経験式、簡略化した置き換え表現を指す。

この文書でいう `0D CAE` とは、空間分布を直接解くのではなく、容積、流量、圧力、温度、トルクなどを
集中定数系として扱う `zero-dimensional` なモデル化と、その上で行う解析、適合、可視化の流れを指す。

これまでのコードには、AI の可能性を探るための試行的、探索的な要素が含まれていた。
これから先の開発は、その探索自体を主目的にはしない。
この方針書に沿って、目的、根拠、制約、検証方法を明確にしたうえで進める。

## 1. 目標

- 物理ベースのエンジンモデルを整備する。
- そのモデルを用いて最適運転点探索と適合を行い、再利用可能な適合マップを生成する。
- 適合済みのマップを実行系で利用し、高速計算とリアルタイム可視化を両立する。
- 数式と実装の対応を明示し、式からコード、コードから式へ辿れる状態を保つ。
- 数値計算の確からしさを評価可能にし、離散化誤差、収束性、安定性を説明できるようにする。
- 数理モデルの各式や各項が挙動へどう効いているかを視覚的に示し、説明可能なシミュレーターにする。
- 利用者がモデル、マップ、制約、車体応答の関係を学べる、教育的なシミュレーターにする。
- 最後には、理解できるだけでなく、触っていて楽しいと思えるシミュレーターにする。
- 数学、物理、工学そのものを面白いと感じられる体験を作る。
- 見た目、動き、反応、情報の見せ方を含めて、かっこいいと思えるシミュレーターにする。
- 開発の最後まで `0D CAE` を主軸とし、より高次の `1D` や `3D` 解析へ拡張することを前提にしない。
- 最終的には `controller` 入力で車体を動かせるインタラクティブなシミュレーターへ発展させる。

## 2. 基本方針

- 重い計算と軽い計算を分離する。
- エンジン側の中核は最後まで `0D CAE` とし、この枠組みの中で適合、可視化、過渡再現を成立させる。
- 物理モデルは、適合と検証の基準となる一次情報として残す。
- 実行系は検証済みのマップを使い、GUI や将来の車体運動を軽く動かす。
- マップは、与えたモデルに対する内部の適合結果として生成し、手作業の断片ではなく再生成可能な成果物として管理する。
- 物理妥当性と数値妥当性は分けて評価し、両方を説明可能にする。
- 可視化は見た目の派手さよりも、物理的意味と判断材料の提示を優先する。
- UI と文書は、利用者の理解を助ける教育的な導線を持たせる。
- 楽しさは軽視せず、理解、操作感、反応の気持ちよさを通じて自然に生まれる形を目指す。
- 楽しさの中心は、数学、物理、工学の関係が見えてくること自体に置く。
- かっこよさは表面的な装飾ではなく、筋の通った情報設計、気持ちのよい反応、統一感のある表現から作る。

## 3. 推奨アーキテクチャ

### 3.1 `0D` モデル・適合系

- 物理ベースの `0D` エンジンモデル
- 運転点探索
- 目的関数と制約条件の評価
- シミュレーター内での適合実行
- 適合マップの生成
- マップの検証と由来記録

### 3.2 実行系

- 検証済みマップの補間
- 範囲外入力の検知と外挿ガード
- リアルタイムのダッシュボードとプロット
- 将来の `controller` 入力と車体応答

### 3.3 説明系

- 現在の運転点の表示
- マップ上の現在地と周辺勾配の表示
- 式、項、`parameter` の感度または寄与度の表示
- 有効な制約条件の表示
- 物理ベースの項と、近似、`surrogate`、数値都合の処理の区別表示
- `timestep`、収束状態、誤差警告、外挿警告などの数値健全性情報の表示

## 4. 段階的な実装順

1. まずエンジンモデルの入出力、単位、制約、検証条件を整理する。
2. 次に最適運転点探索の目的関数と制約条件を定義し、適合ワークフローを作る。
3. その後に適合マップの生成、保存、再読込、補間を実装する。
4. マップ実行系を GUI に統合し、高速表示とインタラクティブな操作を安定させる。
5. 数式の重要度可視化を追加し、挙動の根拠を説明可能にする。
6. 最後に `controller` 入力と車体運動を接続する。

当面は、車両全体を先に広げるよりも、エンジンモデルとマップのワークフローを先に固める。
また、精緻化が必要になっても、まずは `0D CAE` の中で表現力と説明力を高める。

## 5. 暫定の初期判断

現時点では、以下を既定の初期方針とする。詳細は今後の議論で更新する。

- 最初のマップは `steady-state` を中心に、内部のシミュレーション適合で生成する。
- 主軸は `engine speed x requested load` か `engine speed x brake torque` を第一候補にする。
- 最適化の第一目的は、まず `efficiency`、出力、温度余裕のようなエンジン側の指標に置く。
- 制約条件は、まず現在のモデルが表現できる範囲から始める。
- `transient correction` は `base map` を作った後の追加層として扱う。
- `vehicle model` は最初は `longitudinal 1D` に限定する。
- `controller` 入力は、マップ実行系と車体応答が安定してから統合する。
- より高次の `1D/3D` 解析への展開は、現時点では既定路線にしない。

## 6. 数理モデル可視化の方針

- 可視化は `black-box` 的な重要度表示だけで終わらせない。
- できるだけ「どの式のどの項が、どの出力へ効いたか」を追える形にする。
- 数式と実装の対応は、説明文だけに頼らず、追跡可能な対応関係として管理する。
- 初期実装では、局所感度、差分感度、寄与度分解のような説明しやすい手法を優先する。
- 重要度表示は現在の運転点と結び付け、一般論ではなく現在の挙動として見せる。
- 感度計算の方法、差分幅、対象出力は UI 上または文書上で明示する。
- 「物理式そのもの」「物理に基づく簡略化」「経験的 `surrogate`」「数値安定化のための処理」を区別して見せる。
- 数値的な限界や不確かさを隠さず、必要なら警告や注記として利用者に示す。

## 7. 教育性の方針

- 機能追加では、動けばよいだけでなく、学習者が意味を追えることを重視する。
- UI は現在値だけを並べるのではなく、入力、内部状態、出力、制約の関係を理解しやすく示す。
- 複雑なモデルや適合ロジックには、必要に応じて式の意味、前提、近似の理由を説明する導線を付ける。
- 学習者が「この挙動はどの式に基づくのか」「どこからが近似なのか」「どこが物理そのものではないのか」を辿れるようにする。
- 数学、物理、工学のつながりが見えるように、式、現象、設計判断の対応をなるべく切らさない。
- 可視化は、誤解を招く演出よりも、因果関係や物理的解釈の伝達を優先する。
- 用語、単位、軸、制約条件は、学習用途でも読み違えにくい表現にする。
- 教育的価値を損なう強すぎる抽象化や隠蔽は避け、必要なら簡略化の理由と影響範囲を明示する。

## 8. 楽しさの方針

- 最終的な体験として、理解できることと同じくらい、触っていて楽しいことを重視する。
- 楽しさは、誇張した演出だけで作るのではなく、操作に対する納得感のある反応、見て分かる変化、試したくなる余地から作る。
- 楽しさの中心は、数学、物理、工学がつながって動いていると実感できることに置く。
- 可視化、音や動き、操作系を追加するときは、物理的意味や教育性を壊さない範囲で楽しさを高める。
- 学習用だから地味であるべき、という前提は置かず、理解と没入感が両立する設計を目指す。
- `controller` 入力や車体応答の導入では、正確さだけでなく、触ったときの手応えや一貫した反応も評価対象にする。

## 9. かっこよさの方針

- 最終的な体験として、かっこいいと思えることを大事にする。
- かっこよさは、派手な演出の量ではなく、統一感、密度、反応の気持ちよさ、情報の整理の良さから作る。
- 数学、物理、工学の内容がそのまま見た目の魅力や動きの魅力につながる設計を目指す。
- UI、可視化、音や動きは、意味のある変化が美しく見えるように整える。
- 無意味に装飾を増やすのではなく、必要な情報が引き締まって見え、触りたくなる画面を目指す。
- かっこよさは教育性や説明性と対立させず、むしろ理解が深まるほど魅力が増す形を目指す。

## 10. 数値計算の確からしさの方針

- 物理モデルが妥当でも、数値積分や補間が不適切であれば、その結果は信頼しない。
- `solver`、`timestep`、補間、外挿、`clamp`、収束判定は、挙動への影響を検証してから採用する。
- 精度、安定性、計算速度の `trade-off` は、暗黙で決めず、選定理由を残す。
- `timestep` 依存性、離散化依存性、初期値依存性、反復収束性を確認できるようにする。
- 数値安定化のための処理を入れる場合は、何を防ぎ、何を歪めうるかを説明する。
- `extrapolation` や `fallback` に入ったときは、正常系と同じ顔をさせず、検知可能にする。
- 参照解、解析解、単純化した検証問題、既知傾向との比較を用いて数値健全性を確認する。

## 11. コーディングルール

### 11.1 設計原則

- 物理モデル、適合ロジック、実行系マップ、可視化は責務を分離する。
- `0D CAE` の範囲で完結する責務分割を優先し、高次元解析を前提にした構造へ先回りしない。
- GUI は `app orchestration`、runtime `state`、display 用 `view model`、panel / widget の責務を分け、巨大な単一ファイルや単一 struct への集中を避ける。
- GUI の panel は simulator 内部状態へ直接深く触りすぎず、表示用に整形した読み取り専用の情報を主に受け取る構造を優先する。
- 複雑な plot や custom drawing は専用 `module` に閉じ込め、可視化の都合による処理と物理モデル本体を混在させない。
- 新しい `module` には入力、出力、単位、有効範囲を明示する。
- 実行系がマップを使う場合でも、元になったモデルと生成条件を追跡できるようにする。
- 主要な式、`closure`、マップ、`surrogate` には、実装と対応付けるための安定した識別子または参照手段を持たせる。
- 根拠不明の `hard-coded` な補正値を避け、必要なら根拠か調整理由を残す。
- 物理的でない処理、近似処理、暫定補正、数値安定化処理は、物理モデル本体と区別して命名、配置、文書化する。
- 数値パラメータは物理パラメータと混在させず、`solver` 設定や補間設定として分離する。

### 11.2 再現性

- 適合実行時は設定、目的関数、制約、`seed`、使用モデルの版を記録する。
- マップは再生成手順を持つ成果物とし、手編集を前提にしない。
- 補間、外挿、`clamp` の挙動はコードと `test` で固定する。
- 数値実験では `solver` 種別、`timestep`、反復条件、許容誤差、補間設定も記録する。

### 11.3 検証

- 新しいマップ経路には、単位整合、範囲外入力、補間結果、`fallback` の `test` を付ける。
- 最適化は、収束失敗時の扱いと `failure mode` を明示する。
- 可視化は、表示値の算出元を追える状態にしておく。
- 物理式と一致する部分、`surrogate` に頼る部分、数値都合の補正を掛ける部分を `test` や文書で区別する。
- `solver` や補間器を追加するときは、少なくとも `timestep` 感度、収束確認、既知ケース比較のどれかで確からしさを確認する。
- 数値安定化や `clamp` が発火する経路には、発火条件と影響範囲を確認する `test` か診断を付ける。

## 12. 資料利用と知財・秘匿配慮

- 使う資料は、公開情報、公的資料、正規ライセンスの文献、自作データを原則とする。
- 企業秘密、国家機密、流出資料、契約上利用不可の資料、出所不明の内部資料は使わない。
- 秘密保持義務のある情報や、第三者が勝手に公開した資料を正当化して利用しない。
- 外部資料を使うときは、出典、入手日、利用条件、どこへ反映したかを残す。
- 論文、書籍、マニュアルの内容は、自分たちの実装に必要な形へ要約、再構成して使い、長文を無断転載しない。
- 図、UI、意匠、ラベル、配色、レイアウトは既存製品の模倣を避け、独自表現にする。
- 第三者や `vendor` 固有の非公開 `calibration map`、制御戦略、制御ロジックを推測や流用で再現する方向へ寄せない。
- 法的に曖昧な資料は、確認が取れるまで設計入力に使わない。

## 13. AI 利用の法的・契約的ルール

- AI を使う場合でも、適用される法令、規制、契約、利用規約、ライセンスを優先し、AI 出力を免責理由にしない。
- AI に入力する情報は、外部送信してよい情報だけに限定し、個人情報、秘密情報、契約上持ち出せない情報を無断で入れない。
- AI が生成したコード、文章、図、仕様案は、そのまま採用せず、著作権、ライセンス、秘匿性、出典妥当性を確認してから使う。
- 特定製品のコード、資料、図面、UI、文言の模倣や、権利侵害の疑いがある再現要求には寄せない。
- AI 出力に法的判断が絡む場合は、断定で進めず、公式文書、契約文面、公開ライセンス条件で裏取りする。
- AI 利用によって作った成果物でも、最終判断責任は人が持つ前提でレビューと記録を残す。
- 法的または契約的に不明確な点がある場合は、実装を先に進めるよりも、利用可否の確認を優先する。

## 14. 実装判断の優先順位

優先順位は次の順に置く。

1. 追跡可能性 (`traceability`)
2. 再現可能性 (`reproducibility`)
3. 物理解釈性 (`physical interpretability`)
4. 数値的信頼性 (`numerical credibility`)
5. 教育的明瞭性 (`educational clarity`)
6. 楽しさ (`fun`)
7. かっこよさ (`coolness`)
8. 実行時安定性 (`runtime stability`)
9. 可視化品質 (`visualization quality`)
10. 機能拡張 (`feature expansion`)

見た目や機能数を増やす前に、モデルとマップの根拠を固める。

## 15. 今後の議論で決める項目

以下は、次の会話で優先的に詰めたい論点である。

1. 最適化の目的関数を何に置くか。燃費、出力、温度余裕、`lambda = 1` を優先する運転のどれを第一に置くか。
2. マップの主軸を何にするか。`speed x load`、`speed x torque`、`speed x throttle` のどれを基準にするか。
3. 制約条件をどこまでモデル化するか。`lambda`、`cylinder pressure`、`temperature`、`knock surrogate`、`emission surrogate` をどう扱うか。
4. 数式と実装をどう紐づけるか。式 ID、コード `annotation`、対応表、UI 連携のどれを基盤にするか。
5. 「物理的な部分」と「非物理または近似の部分」をどう分類し、どう表示するか。`surrogate`、`clamp`、補正、補間、数値安定化をどう扱うか。
6. 数値計算の確からしさをどう評価するか。`timestep sweep`、`solver` 比較、既知解比較、保存量監視のどれを基準化するか。
7. 数式の重要度表示を何で始めるか。局所感度、項別寄与、`parameter sweep` のどれを第一段階にするか。
8. `vehicle model` の初期スコープをどこまでにするか。`longitudinal 1D` のみか、`gear shift` や `tire slip` まで入れるか。
9. `controller interface` を何にするか。抽象入力、`keyboard/gamepad`、実機 `controller` のどこから始めるか。
10. 楽しさをどこで作るか。操作感、可視化、音や動き、達成感のどれを優先するか。
11. かっこよさをどこで作るか。画面構成、可視化、動き、音、操作感のどこを核にするか。

## 16. 運用ルール

- 実装方針を変える変更は、この文書か関連文書の更新を伴わせる。
- 大きい設計変更を始める前に、「この変更はモデル、適合、実行系、可視化のどこを良くするのか」を確認する。
- 大きい UI や可視化の変更では、「利用者の理解を本当に助けるか」を確認する。
- 体験に関わる変更では、「それは本当に楽しいか」「楽しさが説明性や一貫性を壊していないか」を確認する。
- 見た目や演出に関わる変更では、「それは本当にかっこいいか」「かっこよさが意味や一貫性を壊していないか」を確認する。
- AI を使って設計案や実装案を作った場合も、この文書の知財、秘匿、法的ルールに反していないか確認する。
- `solver` や数値設定を変える変更では、精度、安定性、計算速度のどれにどう効くかを確認する。
- 迷ったときは、「将来マップを再生成できるか」「出典と根拠を説明できるか」を判断基準にする。
